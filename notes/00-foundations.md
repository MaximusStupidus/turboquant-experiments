## 1. Attention

**What is attention doing, conceptually? What do Q, K, and V each represent — and is there any difference in how they get used, even though they're computed the same way?**

Attention is a sequential concept. To truly understand semantic meaning, one needs to be able to look at what came before and only then can they predict what should come next.

> [Note : attention itself is not inherently sequential — in BERT-style models it looks both forward and backward. The "only look backward" part comes from a causal mask that we apply on top of attention in decoder-only models like Llama.]

Attention helps models do exactly this. It works due to three major pillars called Query, Key and Value. Say that we are in a sentence on position n and the n+1th word has to be predicted.

> [Note : Attention itself doesn't predict the next word. Attention produces a contextualized representation of the current token (a blended summary). That representation then gets passed through more layers, and eventually a final layer produces the next-word prediction. Attention is the information-gathering step, not the prediction step.]

So first, using a learned matrix `W_Q`, we will produce the Query vector for the nth embedding. This query vector is like asking: "What information should I focus on to understand the context of the current word". The Key vector is generated by a learned matrix `W_K` and the Value vector by `W_V`.

> [Note : It would be super dumb to think that the Key and Value vectors are computed repeatedly in every step. They are computed once and stored in memory, hence the "KV Cache" and not a "KV Buffer". What's important to understand is that `K_i` and `V_i` vectors are basically independent of the current token — they are computed and stored and remain the same for every future step.]

The dot product of the Query and Key matrices, normalised and softmaxed, represent the answer to the question "What should we focus on". They are represented as `softmax(QK^T / sqrt(d))` and this entire thing is then multiplied by the Value vector to get the attention output. It is important to note that to make our operations efficient, we need to store the K and V vectors somehow in memory so as to not recompute them each time.

> [Note : Saying that the KV Cache makes our operations efficient is underselling its importance. It does much more and is absolutely critical, not a nice-to-have optimisation. Without it, generating a sequence of length n would take O(n^2) redundant work per layer per token, which would compound into making long context generation completely impractical.]

## 2. KV cache, autoregressive TTS, and memory math

**Quick aside on TTS: doesn't TTS just convert textual embeddings into wave embeddings?**

This is how old TTS systems worked. Today's neural TTS systems work differently. They are "autoregressive", which basically means they predict the next chunk of audio, one chunk at a time, in exactly the same way an LLM predicts the next token.

Around 2022–2023, people trained a NN called a "neural audio codec" (e.g. EnCodec, DAC) whose job is to take a chunk of raw audio waveform and compress it into a small sequence of discrete integer tokens (similar to how a JPEG encoder compresses images), and using that same codec, convert these integer tokens back into audio waveforms with little quality loss.

Then, treat audio as a language. Once the audio is a sequence of integer tokens, we can train language models on top of it. VibeVoice (the model we use in part 2) is exactly this: a transformer language model whose vocabulary is audio tokens, with attention, and therefore with a KV cache that is structurally identical to Llama's. That's why TurboQuant is even applicable to it.

### How large is the KV Cache in terms of memory?

Let:
- `n` = number of layers
- `l` = sequence length
- `b` = bytes per number
- `d` = dimension of head
- `h` = number of KV heads

For Llama-3.1-8B: `n = 32`, `h = 8`, `d = 128`, `b = 2` (fp16).

At 8000 tokens, it is roughly:

```
Memory = 8000 * 2 * 32 * 8 * 128 * 2  ≈  1.04 × 10^9 bytes  ≈  1 GB
```

**General formula:**

```
KV cache memory  =  2 × B × S × L × n_kv_heads × d_head × P_bytes
```

where:
- `2` accounts for storing K and V both
- `B` = batch size (number of sequences processed in parallel)
- `S` = sequence length (prompt + generated tokens so far)
- `L` = number of transformer layers
- `n_kv_heads` = number of Key/Value attention heads
- `d_head` = dimension of each attention head
- `P_bytes` = precision in bytes per parameter (FP32 = 4, FP16/BF16 = 2, INT8 = 1)

So, anything that compresses the KV cache (like TurboQuant) directly translates into either:
1. **Longer context windows** on the same hardware, or
2. **More concurrent users** on the same hardware, or
3. **Cheaper hardware** for the same workload.

## 3. Quantization

**1. What is quantization and why does it work on neural networks? 2. What's specifically harder about quantizing the KV cache compared to quantizing model weights?**

Quantization is essentially reducing the precision while representing a mathematical quantity to serve the purpose of memory savings. Say we have to describe an apple. We call it a red coloured fruit. We don't say that it's a hot red coloured protein compound. It's basically getting into less detail because the purpose of communication is served well by less description.

In the context of NNs, quantization post-training works well because the network learns things in abstract dimensions. If you perturb a few weights, the learnings of those few weights are also spread across other weights which compensate for the perturbations. Another way of interpreting this is via LLN, which basically describes how the model approaches its ideal learning state despite minor fluctuations due to the large number of parameters and operations in the model.

> [Note : the law of large numbers says that when you sum a lot of independent random quantities, the random parts tend to cancel out and the sum gets closer and closer to its expected value. Apply this to a neural network: when a layer multiplies a weight matrix by an activation vector, the output of each neuron is a sum of thousands of (weight × activation) products. If you add a tiny independent random error to each of those products via quantization, those errors are zero-mean (equally likely to be too high or too low), and when you sum thousands of them, they mostly cancel each other out. The neuron's output ends up only slightly wrong, even though every individual product was a little wrong.]

Now, specifically for the KV cache, it is a subtype of activation quantization and not weight quantization. Weight quantization is easy because it can be done once and we are through, but activation-layer quantization has to be done on every forward pass and the activations change with the inputs. Manipulating them without careful analysis can severely degrade the model because not all of them affect the model's capability with the same sensitivity.

> [Note : weights are static and known in advance — you can spend hours analyzing the weight distributions, finding outliers, designing custom scales for different layers, even doing iterative refinement. You have all the time in the world. Activations don't give you that luxury — they're produced fresh for every input, and you have milliseconds to compress them before they go into the cache. So you need a quantization algorithm that's both (a) fast enough to run online during inference and (b) good enough to handle whatever statistical structure the activations happen to have on this particular input.]

## 4. Random projections, JL, and TurboQuant

### Johnson–Lindenstrauss Lemma

You can take a set of high-dimensional vectors, multiply each one by a random matrix to project them down into a much lower-dimensional space, and the pairwise distances and dot products between the vectors will be approximately preserved.

### Why JL is relevant to KV cache quantization

JL is relevant to KV cache quantization simply because what the model needs is for the **geometric relationship between the activations** to be preserved — precisely, the dot product between the Key and the Query vectors. So instead of preserving the K and V vectors in their original forms, we only need to focus on the dot products and the final weighted sum, and JL guarantees exactly this.

### The outlier problem and how TurboQuant solves it

Usually, K/V vectors have a few channels with high outlier values (usually 10–100× bigger than typical ones), and this wrecks naive bit-quantization because the bucket budget gets eaten by outliers. Hence, TurboQuant takes a different approach using random projections, which also smooth out these outliers.

TurboQuant is essentially a **2-step process**:
1. **Project** to flatten the distribution and reduce the outliers (the "smear into the soup" effect).
2. **Bit-quantize** the projected values, which are by now Gaussian-friendly.

### Error scaling

The error varies as **1/√k** with the projection dimension k. This implies that the smaller the dimension (the more the compression), the larger the error. Less compression → more fidelity.

Empirically I confirmed this with two pedagogical scripts:
- **`jl_demo.py`** — projected 1024-dim vectors down to k = 4, 8, …, 512 and measured median relative error on pairwise squared distances. At k = 128 (8× compression) the error was ~8.5%; at k = 512 (2× compression) it was ~4%. The measured curve tracked the 1/√k reference line cleanly across two orders of magnitude.
- **`outlier_demo.py`** — generated vectors with 3 outlier channels at ~±40 magnitude, then projected. The original distribution had values reaching ±144 (extreme/typical ratio ~60×); after projection the same vectors had values contained within roughly ±27 (extreme/typical ratio ~5×), with a smooth Gaussian-ish shape. The outlier energy was preserved (it has to be — JL preserves geometry) but redistributed across all the projected dimensions instead of being concentrated in a few channels.

This is the visual evidence that the two TurboQuant ingredients — geometric preservation and distribution flattening — both actually happen on real numerical data, not just in the paper.
