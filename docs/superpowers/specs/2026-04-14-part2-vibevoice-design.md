# Part 2 — TurboQuant on VibeVoice-Realtime-0.5B (Design)

**Status:** Draft
**Date:** 2026-04-14
**Prerequisite:** Part 1 complete (TurboQuant validated on Llama-3.1-8B)

---

## 1. Purpose

Apply TurboQuant KV cache quantization to an **autoregressive TTS model** and measure whether speech quality, voice identity, and latency survive compression. This tests the core thesis from Part 1 in a completely different domain: if TurboQuant works because it preserves dot products in the KV cache (a domain-agnostic property), it should work on any autoregressive transformer — including one that generates audio instead of text.

## 2. Model: VibeVoice-Realtime-0.5B

**Why this model:**
- Open-source (MIT license), weights on HuggingFace, working inference code
- Autoregressive transformer with standard HF `DynamicCache` interface
- Declares `_supports_quantized_cache = True`
- Small enough to run on any GPU (0.5B params, ~1 GB weights)
- Pre-computed voice prompts included in the repo for reproducible testing

**Architecture:**

| Parameter | Value |
|---|---|
| Base model | Qwen2.5-0.5B |
| Total layers | 24 (split: 4 base LM + 20 TTS LM) |
| Q heads | 14 |
| KV heads | 2 (GQA 7:1) |
| Head dim | 64 |
| Hidden size | 896 |
| Max context | 8,192 tokens |
| Audio frame rate | 7.5 Hz (continuous 64-dim latents, not discrete tokens) |
| Diffusion head | 4 layers, 20 steps (not cached — only the LM backbone has a KV cache) |

**Split architecture:** The 24 layers are split into:
- **Base LM (4 layers):** Processes text tokens only. Small cache, grows slowly.
- **TTS LM (20 layers):** Processes text + speech tokens. Large cache, grows fast. **This is our TurboQuant target.**

**4 KV caches in the generation loop:**
- `past_key_values` — Base LM (positive), 4 layers
- `tts_lm past_key_values` — TTS LM (positive), 20 layers
- `neg past_key_values` — Base LM (negative, for CFG), 4 layers
- `neg_tts_lm past_key_values` — TTS LM (negative, for CFG), 20 layers

We target the **TTS LM caches** (positive + negative, 20 layers each) since they accumulate the most tokens and dominate memory at long sequences.

**Correction from Phase 0:** VibeVoice uses **continuous acoustic latents** (64-dim vectors from a VAE), not discrete audio tokens. The model is still autoregressive (generates one position at a time with a KV cache), but each "token" is a continuous vector generated by a diffusion head, not a discrete index from a codebook. The KV cache structure is identical to a standard LLM — TurboQuant applies the same way regardless.

## 3. Metrics

| Metric | What it measures | Tool/Library | Threshold for "working" |
|---|---|---|---|
| **UTMOS** | Audio naturalness (1-5 MOS scale) | `utmos` / SpeechMOS | < 0.3 drop from baseline |
| **Speaker similarity** | Voice identity preservation (cosine sim) | SpeechBrain ECAPA-TDNN | > 0.85 similarity to baseline |
| **WER** | Speech intelligibility (word error rate) | Whisper transcription | < 5% absolute increase |
| **Time-to-first-audio** | Latency until first audio chunk | Wall-clock timing | Informational (no hard threshold) |
| **Real-time factor (RTF)** | Generation time / audio duration | Wall-clock timing | < 1.0 means real-time capable |

**Why these metrics:**
- UTMOS: Does the speech still sound natural? (catches artifacts, distortion)
- Speaker similarity: Does the voice still sound like the same person? (catches voice identity loss — the "does compression change who's speaking?" question)
- WER: Can you understand the words? (catches intelligibility degradation)
- TTFA + RTF: Is it still fast enough for real-time use? (the practical deployment question)

## 4. Experimental configs

Same structure as Part 1 — fp16 baseline, then TurboQuant at multiple bit widths:

| Config | Method | Target |
|---|---|---|
| fp16 (baseline) | No compression | Reference point |
| TurboQuant 4-bit | Handrolled, projection + quantization | Moderate compression |
| TurboQuant 3-bit | Handrolled, projection + quantization | Sweet spot (from Part 1) |
| TurboQuant 2-bit | Handrolled, projection + quantization | Aggressive compression |

We use our **handrolled implementation** (not the community package) because:
- We validated it in Part 1 (matches community PPL within noise)
- It's the educational artifact we're open-sourcing
- We need to adapt `head_dim=64` (vs 128 in Part 1) — easier to adjust our own code
- The community `turboquant` package may not handle the 4-cache split architecture

We skip KIVI for Part 2 (the comparison was already done in Part 1; adding it here would be redundant).

## 5. Test data

**Voice prompts:** Use the pre-computed `.pt` files from the repo (`demo/voices/streaming_model/`):
- `en-Carter_man.pt` — male English voice
- `en-Emma_woman.pt` — female English voice

**Text prompts:** A fixed set of sentences covering different lengths and content:
1. Short: "Hello, this is a test of voice quality."
2. Medium: A paragraph from the repo's example texts.
3. Long: Multiple paragraphs to stress the KV cache at extended context.

Each (voice, text, config) combination produces one audio file. We generate all combinations and measure all metrics on each.

## 7. Key risks

| Risk | Likelihood | Response |
|---|---|---|
| HandrolledTurboQuantCache doesn't work with the 4-cache split | Medium | Debug the cache interface; may need to wrap rather than replace |
| head_dim=64 causes worse quantization than head_dim=128 | Medium | Fewer dimensions = fewer averaging opportunities for JL. If 2-bit fails, 3-bit may still work. |
| Inference code has undocumented dependencies | Low | The repo has a Colab notebook that should list everything needed |
| Audio quality degrades even at 4-bit | Low | Would be a surprising but publishable negative result |
| Voice identity lost at 2-bit | Medium | The speaker embedding is captured in the early cache (voice prompt); if the prompt cache stays fp16 and only generated tokens are quantized, identity should survive |

## 8. Important design decision: what to quantize

The voice prompt's KV cache (loaded from `.pt` files) captures the **speaker's identity**. The generated tokens' KV cache captures the **content being spoken**. We have a choice:

**Option A:** Quantize everything (prompt cache + generated cache). Maximizes compression but risks degrading voice identity.

**Option B:** Keep the prompt cache at fp16, only quantize the generated tokens' cache. Preserves voice identity perfectly, still compresses the growing part of the cache.

**Decision:** Start with **Option B** (safer). If metrics are good, also test Option A to see if voice identity survives. This is a natural ablation that produces a concrete finding: "does speaker identity survive KV cache compression?"
